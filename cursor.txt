学习助手 App 云端同步问题修复需求文档
1. 项目背景
当前学习助手 App 已实现以下功能：

使用 Supabase 作为后端，app_state 表存储用户数据（user_id 主键，data JSONB 字段）。

多端实时同步：通过 Supabase 的实时订阅（Realtime）实现数据变更的即时同步。

切屏/后台恢复：监听 visibilitychange 事件，页面恢复可见时重新订阅，确保同步不中断。

然而，近期发现一个关键 Bug：当用户在一端登录并修改数据（云端已同步成功）后，主动登出账号，再重新登录同一账号时，之前的所有修改全部丢失（多端均显示数据为空）。

2. 问题详细描述
2.1 重现步骤
在手机端登录账号test@example.com

在应用中添加/修改一些数据（例如添加笔记、添加心情）。

确认数据已成功保存到云端（可在另一设备电脑上登录同一账号验证数据已同步）。

在手机端中点击“登出”按钮，退出账号。

等了一会在手机端中重新登录同一账号 test@example.com。

观察应用数据：之前添加/修改的数据全部消失，界面显示为空。

同时检查电脑端（仍在登录状态），发现数据也变为空（或被清空）。

2.2 影响范围
用户数据永久丢失：登出再登录后，原有数据被清空，且无法恢复。

多端一致性被破坏：所有已登录的客户端均被清空。

3. 期望行为
用户登出后，本地内存中的用户数据应被清除（保障隐私），但云端数据必须保持不变。

用户重新登录同一账号时，应用应自动从 Supabase 拉取该账号的最新数据，并正确显示，不应丢失任何先前保存的数据。

4. 可能原因分析
根据现有代码逻辑（useAuth 和 useDataSync Hook），推测问题可能出现在以下环节：

4.1 登出时错误地清除了云端数据
登出操作是否误触发了对 app_state 表的 DELETE 或 UPDATE 操作，导致云端数据被删除？

检查 signOut 函数：它只调用了 supabase.auth.signOut()，理论上不应影响数据库表。但需确认是否有其他副作用（例如在组件卸载时自动保存空数据）。

4.2 重新登录后未正确拉取云端数据
useDataSync 在 user 从 null 变为有效时，会执行 fetchData 从 app_state 拉取数据。但可能存在以下问题：

拉取请求失败（网络错误、权限问题），但代码未处理错误，导致 data 保持为 null。

拉取成功但返回 null（例如表中无该用户记录），但此时应该返回一个空对象 {} 或默认初始数据，而不是 null。如果后续操作（如自动保存）将 null 写入云端，则会覆盖原有数据。

检查实时订阅：重新登录后是否成功建立了新的订阅？如果订阅未正确建立，可能导致后续更新无法同步，但不应导致数据丢失。

4.3 登出/登录过程中出现竞态条件
登出时，useDataSync 的清理函数（取消订阅）是否及时执行？重新登录时，旧订阅未完全清理，新订阅可能未正确建立，导致数据状态混乱。

是否有多个组件同时操作数据，导致状态不一致？

4.4 数据保存时机问题
是否存在某些操作（如页面卸载前）自动保存当前 data 到云端？如果登出前 data 已被设置为空（例如因为登出清空状态），那么自动保存会将空数据上传云端，覆盖原有数据。

检查是否有 useEffect 依赖 data 变化自动保存，或者有 beforeunload 事件处理程序。

6. 修复要求
请对现有代码进行修改，确保登出再登录后数据不丢失。具体要求如下：

6.1 修复登出逻辑
确认 signOut 函数仅执行认证登出，不执行任何数据库写操作。

在登出时，应取消实时订阅（已在 useDataSync 的清理函数中实现），但不要主动清空云端数据。

6.2 修复重新登录后的数据加载
在 useDataSync 中，当 user 从 null 变为有效时，执行数据拉取。需确保：

拉取失败时应有错误提示（console.error），但不应将 data 设置为 null 或默认值，而应保持之前的状态（可能为空），避免自动覆盖云端。

如果拉取结果为空（用户第一次登录，表中无记录），应设置 data 为一个合理的默认值（例如 {} 或 []），但不能是 null。同时，在初始加载时，应避免自动保存默认值到云端（除非用户主动触发保存）。

建议在 fetchData 成功后，使用 setData(record?.data ?? defaultValue)，其中 defaultValue 是符合应用初始状态的对象（例如 { notes: [] }）。若表中无记录，不自动插入默认记录，仅本地状态设为默认值。

6.3 防止自动保存覆盖云端数据
检查所有调用 saveData 的地方：是否有在组件卸载、页面刷新或登出时自动保存当前 data 的逻辑？若有，需移除或增加保护条件（例如仅当用户主动操作时才保存）。

特别是检查 useEffect 依赖 data 的自动保存：如果每次 data 变化都自动保存，那么在登出时 data 被清空（因 user 变为 null 触发 setData(null)），自动保存会将 null 写入云端。应改为：仅当 user 存在且 data 由用户操作改变时才保存。

6.4 增强错误处理与日志
在所有 Supabase 操作（查询、订阅、upsert）中添加详细的错误日志，便于追踪问题。

在控制台输出关键步骤（如“登出”、“重新登录”、“数据拉取成功/失败”），方便调试。

6.5 代码注释
为修改后的关键逻辑添加清晰注释，解释为何如此处理（例如：避免登出时误清空数据、防止自动保存覆盖等）。

7. 验证步骤
修改完成后，请按以下步骤验证：

准备两个浏览器（Chrome 和 Firefox）或同一浏览器的普通模式+隐私模式。

在浏览器 A 登录账号，添加几条测试数据（例如笔记）。

在浏览器 B 登录同一账号，确认数据已同步显示。

在浏览器 A 登出账号，确认界面数据被清空（本地清除）。

在浏览器 A 重新登录同一账号，观察数据是否恢复为步骤 2 中添加的数据。

检查浏览器 B，确认数据依然存在且与步骤 2 一致（未被清空）。

重复多次登录、登出、修改数据，确保每次重新登录都能正确恢复云端数据。

模拟网络中断：在重新登录时断开网络，观察错误处理是否得当（数据不应丢失，待网络恢复后应能正确加载）。
