3. 功能需求
3.1 多端数据同步
用户在同一账号下，通过不同设备（如手机浏览器、电脑浏览器）登录时，数据应实时保持一致。

任何一端的数据变更（增、删、改）应立即反映到其他已登录的客户端。

3.2 登录/登出行为
登录后：自动从 Supabase 拉取该用户的最新数据并显示；若本地有旧缓存，应先清除再加载云端数据，且有数据版本比较逻辑，从云端拉取数据之前，先比较本地版本和云端版本哪个更新，
如果本地版本更新，则自动将最新本地版本数据上传云端同步，保证用户数据是最新的，不无脑云端数据覆盖。

登出后：清除当前网页中的所有用户数据（包括内存和 localStorage 中的状态），确保下一位登录用户不会看到前一位用户的数据。

重新登录：同账号再次登录时，应重新拉取云端数据，恢复之前的内容。

3.3 切屏/后台恢复
用户切换浏览器标签页、最小化浏览器或手机浏览器切到后台再返回时，应用应能自动恢复云端同步连接，不丢失数据，无需手动刷新。

网络短暂中断后重连，同步应自动恢复。

4.2 数据库表结构
假设用户数据存储于 user_data 表，建议结构如下（可根据实际需求调整字段）：

sql
CREATE TABLE user_data (
  id BIGSERIAL PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  content JSONB NOT NULL,  -- 存储具体的学习内容，格式灵活
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- 索引优化查询
CREATE INDEX idx_user_data_user_id ON user_data(user_id);
实时功能启用：需在 Supabase 控制台的 Database > Replication 中为 user_data 表开启复制（Replication），以便实时订阅生效。

4.3 前端代码要求
使用 Supabase JavaScript SDK（v2 版本）进行客户端初始化、认证和实时订阅。

采用 React Hooks（或你根据目前现有的其他框架的等效方式）封装认证逻辑和数据同步逻辑，提高代码复用性。

确保在页面可见性变化时（如切屏）能主动恢复实时连接。

5. 代码修改具体要求
5.1 消除代码冗余
移除重复的 Supabase 客户端初始化代码，统一在一个模块（如 src/lib/supabaseClient.js）中初始化，并从配置文件中读取 URL 和 key。

合并多处认证状态监听逻辑，使用自定义 Hook（如 useAuth）集中管理。

合并多处数据获取和更新逻辑，使用自定义 Hook（如 useDataSync）封装。

5.2 修复 Bug
URL 变更问题：将所有硬编码的 Supabase URL 和 anon key 统一放置到 src/config/supabase.js 中，其他地方通过导入使用。这样，未来修改仓库名称或项目配置时，只需修改这一个文件即可。

登出数据残留：登出时确保清空本地状态（如 React 组件状态、缓存），并取消实时订阅。

切屏同步中断：监听 visibilitychange 事件，在页面恢复可见时重新订阅实时频道（Supabase 客户端通常会自动重连，但可增加主动重连逻辑以增强可靠性）。

错误处理：对 Supabase API 调用进行错误捕获，并在控制台输出有意义的错误信息，便于调试。

7. 测试验收标准
登录同步测试：在设备 A 登录账号，添加一条数据；在设备 B 登录同一账号，应立即看到新数据。

数据变更同步：在设备 A 修改数据，设备 B 应自动更新。

登出清除测试：登录后登出，刷新页面，不应看到任何用户数据；重新登录后数据恢复。

切屏恢复测试：在设备 A 打开应用，切换到其他标签页一段时间后返回，继续操作数据，设备 B 应能收到更新。

错误处理测试：故意断开网络，然后重连，观察同步是否恢复；控制台不应出现未捕获的致命错误。

配置修改测试：手动修改 src/config/supabase.js 中的 URL 为无效值，应用应能提示错误而不是挂起；恢复正确值后应正常工作。
