1. 版本概述

版本号：v2.0
版本名称：云端同步范围优化
目标：优化现有应用的云端同步机制，确保仅同步用户核心数据（增删改），不同步无关的 UI 状态，提升多端使用体验和同步效率。

2. 背景与问题

2.1 当前现状

学习助手 App 使用 Supabase 作为后端数据库和实时同步服务，实现了多客户端（手机、电脑）之间的数据同步。目前，用户在任一端的操作（包括页面切换、文件夹折叠/展开、标签页激活等）都会被同步到其他客户端，导致不必要的界面跳转和混乱。

2.2 存在的问题

· UI 状态被同步：例如在手机端从“今日任务”页面切换到“笔记”页面，电脑端会自动跳转到“笔记”页面，干扰用户当前操作。
· 同步数据冗余：将 UI 状态（如当前激活的页面、折叠状态）存储在数据库（如 user_settings 表）中，增加了不必要的数据库读写和实时消息量。
· 用户体验下降：多端同时操作时，界面频繁自动跳转，让用户感到困惑和控制权被剥夺。

2.3 用户反馈

用户期望各设备间的页面导航保持独立，仅学习数据（笔记内容、计时记录、学习计划等）能够实时同步，避免互相干扰。

3. 需求目标

· 核心原则：云端同步仅限于用户数据的创建、更新、删除操作（即直接影响数据库记录的变化）。
· 不同步范围：所有 UI 状态（包括但不限于当前页面、折叠/展开状态、滚动位置、临时选中状态、表单输入中的内容等）均不进行跨设备同步。
· 保持数据一致性：用户数据的跨设备实时同步仍需保持，确保在任何设备上对笔记、计时、计划等内容的修改能立即反映到其他设备。
· 本地状态保留：UI 状态可保存在本地（如 localStorage、sessionStorage 或内存中），用于恢复用户上次的浏览位置（仅限当前设备），但不上传至云端。

4. 详细功能要求

4.1 识别并区分数据同步源

· 对现有代码中所有触发 Supabase 数据库写入的操作进行全面审查，标记出那些不应同步的操作（例如：记录当前页面的用户设置、记录文件夹展开状态等）。
· 分类如下：
  · 应同步的数据：笔记内容（增删改）、学习计时记录、学习计划目标、任务完成状态、统计数据、笔记列表的排序方式等直接影响学习过程的数据。
  · 不应同步的数据：当前激活的页面路由、文件夹的展开/折叠状态、滚动条位置、临时输入的草稿（未保存）、模态框打开状态等。

4.2 移除 UI 状态的云端存储

· 若现有代码中将 UI 状态存储到了数据库（例如 user_preferences 表中存储了 last_visited_page 或 expanded_folders），需：
  · 删除相关数据库表字段或整个表（如果不再需要）。
  · 修改前端代码，不再对这些状态进行数据库写入操作。
  · 改为使用本地存储（localStorage）或状态管理库（如 Redux、Zustand、React Context）在单设备内持久化这些 UI 状态，以便用户刷新页面后恢复，但不跨设备共享。

4.3 优化实时订阅机制

· 确保 Supabase 的实时订阅（Realtime）仅监听那些真正需要同步的数据表（如 notes、study_sessions、daily_plans 等），避免订阅包含 UI 状态的表（若之前存在）。
· 在前端，实时订阅的回调中，只更新对应的数据状态，不应触发任何路由跳转或 UI 折叠状态的变更。

4.4 登录/登出时的数据清理

· 登出时，除了清除用户认证信息，还应：
  · 取消所有实时订阅。
  · 清除本地存储中的用户数据（笔记、计时等）。
  · 保留本地 UI 偏好（如是否记住上次页面）可根据产品策略决定是否保留，但必须确保这些偏好不上传云端。
· 重新登录后，仅拉取用户核心数据，UI 状态应恢复到该设备默认值或从本地存储恢复（不上云）。

4.5 多端独立导航

· 各设备的路由导航完全独立，不应受其他设备导航操作的影响。
· 例如：设备 A 从“今日任务”跳转到“统计”页面时，设备 B 的当前页面不应发生任何变化。

5. 技术实现要点

5.1 前端架构调整

· 状态分层：
  · 数据状态：从 Supabase 同步的笔记、计时等，使用全局状态管理（如 Redux、Zustand）并响应实时订阅更新。
  · UI 状态：页面路由、折叠面板等，使用本地组件状态或 Context，不同步至云端。
· 路由管理：确保路由变化不触发数据库写入。若之前有通过 useEffect 监听路由变化并写入 last_page 到数据库，需移除。
· 数据状态管理：笔记列表的排序方式作为用户偏好，应存储在云端（如 user_preferences 表），并通过 Supabase 实时同步，确保多端一致。
· 本地存储策略：对于其他无需同步的 UI 偏好（如临时展开状态），仍可使用 localStorage 存储；排序方式不再归入此类。

5.2 数据库调整（如有必要）

· 检查现有数据库中是否存在专门用于存储 UI 状态的表或字段，例如：
  · user_settings 表中的 last_page、expanded_folders 等字段。
  · 若存在，需评估是否可以删除这些字段。如果仍有本地存储需求，可保留但不再通过 Supabase 同步（即前端不再读写这些字段，或改为仅写入但不订阅）。
· 建议：彻底移除这些字段，避免后续误用。

5.3 实时订阅优化

· 在 Supabase 客户端初始化时，明确只订阅必要的数据表。
· 示例：
  ```javascript
  // 仅订阅笔记和计时表
  const notesSubscription = supabase
    .channel('notes-changes')
    .on('postgres_changes', { event: '*', schema: 'public', table: 'notes', filter: `user_id=eq.${user.id}` }, handleNotesChange)
    .subscribe();
  ```
· 避免订阅整个数据库或包含 UI 状态的表。

5.4 同步触发条件

· 修改数据写入逻辑，确保只有用户主动保存/修改数据时才调用 Supabase API。
· 例如：重命名笔记后调用 supabase.from('notes').update()，而切换页面时不调用任何数据库更新。

6. 数据模型变更（可选）

如果现有数据库中存在以下表/字段，建议在本次版本中移除：

· 表名 user_settings，字段 last_page：记录上次访问的页面，应移除。
· 表名 user_settings，字段 expanded_folders：记录文件夹展开状态，应移除。
· 如果 user_settings 表中已有或需要新增存储排序方式的字段（如 note_sort_order），则不应移除，且需纳入实时订阅范围。
· 其他类似存储 UI 状态的表：根据代码审查结果确定。

注意：移除字段前需确保前端不再依赖这些字段，且数据迁移方案（如有历史数据需要保留）需谨慎处理。若暂时不能完全移除，可先停止写入和订阅，后续再清理。

7. 测试验收标准

7.1 功能测试

· 测试场景：页面导航独立性
  · 操作步骤：
    1. 在设备 A 登录，进入“笔记”页面
    2. 在设备 B 登录（同一账号），当前在“今日任务”页面
    3. 在设备 A 切换到“统计”页面
  · 预期结果：设备 B 应保持在“今日任务”页面，不跳转
· 测试场景：数据同步正常
  · 操作步骤：
    1. 在设备 A 添加一条新笔记
    2. 查看设备 B 的笔记列表
  · 预期结果：设备 B 应自动显示新笔记（无论当前在哪个页面）
· 测试场景：折叠状态不同步
  · 操作步骤：
    1. 在设备 A 展开某个文件夹
    2. 查看设备 B 的相同文件夹
  · 预期结果：设备 B 的文件夹应保持原有折叠/展开状态，不受影响
· 测试场景：登出后数据清除
  · 操作步骤：
    1. 设备 A 登录并产生数据
    2. 登出
    3. 重新登录
  · 预期结果：登出后页面应无用户数据；重新登录后数据恢复，UI 状态恢复为默认或本地存储值（但不同步自云端）
· 测试场景：实时订阅有效性
  · 操作步骤：
    1. 设备 A 和 B 同时登录
    2. 在设备 A 删除一条笔记
  · 预期结果：设备 B 的笔记列表应实时移除该笔记
· 测试场景：排序方式跨设备同步
  · 操作步骤：
    1. 在设备 A 登录，将笔记列表排序方式改为“降序”。
    2. 在设备 B 登录同一账号，查看笔记列表排序方式。
  · 预期结果：设备 B 的笔记列表也应自动变为“降序”排序，无需手动设置。

7.2 性能与稳定性测试

· 同步流量：监控 Supabase 的实时消息量，确保较之前有明显下降（因移除了 UI 状态同步）。
· 网络波动：模拟网络断开后重连，确保数据同步恢复，UI 状态不受影响。
· 并发操作：多端同时对同一数据修改，验证冲突处理机制（如最后写入胜出或提示用户）。

8. 注意事项

· 代码审查：需仔细审查所有数据库写入操作，确保没有遗漏的 UI 状态同步。
· 兼容性：如果旧版本客户端仍在使用 UI 状态同步，需考虑升级后的兼容性问题。建议强制用户更新到 v2.0 版本，或后端兼容处理。
· 用户体验：移除跨设备页面同步后，用户可能期望某些 UI 偏好（如暗色模式）能同步，这类偏好应归类为“用户设置数据”而非纯 UI 状态，可考虑后续版本单独实现（但不在本需求范围内）。
· 本地存储策略：若使用 localStorage 存储 UI 偏好，注意容量限制和隐私（不要存储敏感数据）。
· 回滚计划：准备回滚方案，以防上线后出现重大问题。
